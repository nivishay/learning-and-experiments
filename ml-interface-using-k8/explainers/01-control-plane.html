1. Cluster anatomy

A Kubernetes cluster is a set of machines split into two roles: the control plane and worker nodes.

  • The control plane is the brain. It holds all desired state, makes scheduling decisions, and runs the control loops that keep actual state matching desired state. It never runs your application containers.
  • Worker nodes are the muscles. They run your Pods — the actual containers. Each node runs two Kubernetes agents that talk back to the control plane.
Key insight
etcd is the only database. All cluster state — every Pod, every Deployment, every Service — lives in etcd as key-value pairs. No other component stores any state of its own. This is why you can restart any control plane component and it picks up exactly where it left off.
→ kubernetes.io — Cluster Components

2. The four control plane components

kube-apiserver

The only door into the cluster. Every kubectl command, every controller, every kubelet — all talk to the API server over HTTPS. It validates objects, enforces authorization (RBAC), and persists state to etcd. Nothing else reads or writes etcd directly.

etcd

A distributed, consistent key-value store. Holds the entire desired and current state of the cluster. Typically 3 or 5 instances for high availability (uses the Raft consensus protocol). Losing etcd without a backup means losing the entire cluster state.

kube-scheduler

Watches for Pods that have no Node assigned. For each such Pod it scores all available Nodes (taking into account resource requests, node capacity, affinity rules, taints/tolerations) and writes the chosen Node back to the API server. It does not start the Pod itself.

kube-controller-manager

A single binary that runs many controllers in parallel. Each controller watches the API server for changes to objects it cares about and takes action to reconcile actual state toward desired state. Examples: the ReplicaSet controller, the Deployment controller, the HPA controller, the Node controller (evicts Pods from dead nodes).

Why combine all controllers into one binary?
In theory, each controller could be a separate process. In practice, combining them into kube-controller-manager reduces operational complexity. Separating them is possible and used in some custom setups, but the default binary is one process with many goroutines running in parallel.
→ kubernetes.io — Controller pattern

3. The two worker node agents

kubelet

An agent running on every worker node. It watches the API server for Pods assigned to its node. When one appears, it instructs the container runtime (typically containerd) to pull the image and start the container. It also runs liveness and readiness probes and reports Pod status back to the API server.

kube-proxy

Also runs on every node. It watches the API server for Service and Endpoint objects and updates the node's iptables (or ipvs) rules accordingly. This is how a Service's virtual IP gets routed to actual Pod IPs. It does not proxy network packets at the application layer — it sets up kernel-level rules and then gets out of the way.

Try this
On your Minikube cluster, SSH in and look at the running processes: 
minikube ssh
ps aux | grep kubelet
ps aux | grep kube-proxy
You'll see both processes running. Notice kubelet has a --config flag pointing to its config file, and various flags that tell it which API server to talk to.

4. The reconciliation loop — the core mental model

Every controller in Kubernetes follows the same pattern. It is sometimes called the control loop or reconciliation loop:

loop:
  desired  = read desired state from API server (etcd)
  actual   = observe current state of the world
  if desired != actual:
      take action to move actual toward desired
  sleep briefly, repeat

This is the single most important concept in Kubernetes. It is what makes the system self-healing:

  • You delete a Pod manually → the ReplicaSet controller notices the actual count is below desired → it creates a replacement Pod. You never told it to. It just noticed and fixed it.
  • A node crashes → the Node controller marks it NotReady → after a timeout, Pods on that node are evicted and rescheduled on healthy nodes.
  • CPU spikes above 60% → the HPA controller notices → it writes a higher replicas count to the Deployment → the ReplicaSet controller creates new Pods.
Kubernetes is declarative, not imperative
You tell Kubernetes what you want, not how to achieve it. The reconciliation loops figure out the "how". This is the fundamental reason why Kubernetes manifests look like descriptions of state (replicas: 2) rather than commands (start 2 pods).
→ kubernetes.io — Controller pattern
NVIDIA interview angle
At NVIDIA's scale, this matters enormously. If a GPU node fails mid-training, Kubernetes detects the node failure, reschedules the workload, and maintains the declared replica count — without human intervention. Understanding the reconciliation loop is what lets you reason about why Kubernetes is resilient, not just that it is.

5. Step-by-step: what happens when you run kubectl apply -f deployment.yaml

Let's trace your ml-inference Deployment through the entire system. Click each step to expand the detail.

1

kubectl serializes the YAML and sends it to kube-apiserver over HTTPS

kubectl reads your YAML file, converts it to JSON, and sends a POST (if new) or PATCH (if updating) request to the API server at something like https://<cluster-ip>/apis/apps/v1/namespaces/default/deployments. Your ~/.kube/config file holds the cluster address and your credentials.
2

kube-apiserver validates and persists the Deployment to etcd

The API server first authenticates you, then checks RBAC (are you allowed to create Deployments?), then validates the object (are all required fields present? are the values legal?). If everything passes, it writes the Deployment object into etcd. It also sets a unique resourceVersion on the object — a monotonically increasing number used for optimistic concurrency.
3

The Deployment controller (inside kube-controller-manager) is notified

All controllers maintain a watch on the API server for their resource type. A watch is a long-lived HTTP connection that streams events (Added, Modified, Deleted) as they happen in etcd. The Deployment controller sees the new Deployment event and wakes up.
4

The Deployment controller creates a ReplicaSet

The Deployment controller compares desired state (your Deployment spec) with actual state (no ReplicaSet exists yet). It creates a ReplicaSet object via the API server, which is persisted to etcd. The ReplicaSet has a Pod template copied from the Deployment, and replicas: 2. The Deployment controller itself stops here — it has done its job.
5

The ReplicaSet controller creates two Pod objects

The ReplicaSet controller watches for ReplicaSet objects. It sees the new one, counts matching Pods (zero), and creates two Pod objects via the API server. These Pods exist in etcd but have no nodeName set — they are not yet scheduled to any node. They are in the Pending phase.
6

kube-scheduler assigns each Pod to a Node

The scheduler watches for Pods with nodeName == "". For each unscheduled Pod it runs a two-phase algorithm:
  • Filter: eliminate Nodes that can't run the Pod (not enough CPU/memory to satisfy requests, taints that the Pod doesn't tolerate, etc.)
  • Score: rank the remaining Nodes by various criteria (available resources, pod affinity, spread constraints)

It then writes the chosen Node name into the Pod's spec.nodeName field via the API server. The scheduler does nothing else — it does not contact the node directly.
7

The kubelet on the chosen Node starts the container

Each kubelet watches for Pods with its own nodeName. When it sees its Pod, it:
  1. Tells the container runtime (containerd) to pull the image (in your case: ml-interface:v1, from local cache since imagePullPolicy: Never)
  2. Creates and starts the container with the specified resource limits
  3. Begins running the liveness and readiness probes on their configured intervals
  4. Reports Pod status back to the API server (phase transitions: Pending → Running)
8

kube-proxy updates iptables rules for your Service

When you also applied service.yaml, the API server created an Endpoints object that tracks which Pod IPs are currently Ready. kube-proxy watches Endpoints and updates the node's iptables rules so that traffic sent to the Service's ClusterIP gets load-balanced to Pod IPs. Once your readiness probe returns 200, the Pod IP appears in Endpoints and traffic begins flowing to it.
The key pattern
Notice that no component directly commanded another component. kubectl wrote to etcd (via API server). Each controller woke up because it noticed something changed. The kubelet woke up because it noticed a Pod assigned to it. This is level-triggered, watch-based architecture — components react to state, not to messages. It's why the system recovers gracefully from crashes: when a component restarts, it re-reads the current state and picks up from there.
→ Kubernetes blog: The mechanics of Kubernetes

6. The watch mechanism — how components get notified

Every controller uses the API server's watch API. Under the hood:

GET /apis/apps/v1/deployments?watch=true
# → returns a long-lived HTTP chunked response
# → each chunk is a JSON event: {"type": "ADDED", "object": {...}}
#                              {"type": "MODIFIED", "object": {...}}
#                              {"type": "DELETED", "object": {...}}

The API server translates etcd watch events into these HTTP events. Every controller, every kubelet, runs such a watch. When the connection drops (network blip, restart), the client reconnects and uses its last known resourceVersion to get any events it missed.

Try this — watch in real time
Open two terminal windows. In the first: 
kubectl get pods -w
In the second, delete a Pod: 
kubectl delete pod <pod-name>
Watch the first terminal: you'll see the Pod transition through Terminating, then a new Pod appear through Pending → ContainerCreating → Running. That is the reconciliation loop executing in real time.

Quiz — test your understanding

Check yourself

1. You manually delete a Pod that belongs to a Deployment. Who creates the replacement Pod?

Wrong. The scheduler's job is to assign Pods to Nodes — it doesn't create Pods. Pods are created by controllers.
Correct! The ReplicaSet controller runs the reconciliation loop: it watches actual Pod count vs desired, and when they differ, it creates (or deletes) Pod objects via the API server. The scheduler then assigns them to Nodes, and the kubelet starts them.
Wrong. The kubelet handles containers on its own node. If the whole Pod is deleted (not just a container crash), the Pod object is gone from etcd and the kubelet stops tracking it entirely.
Wrong. The API server is passive — it stores and validates objects, it doesn't act on them.

2. What is the single source of truth for all cluster state?

Wrong. The API server does maintain a cache for performance, but it is derived from etcd and is not authoritative. If the API server restarts, it rebuilds its cache from etcd.
Wrong. Kubelets observe local reality and report to the API server, but the desired state lives in etcd.
Correct! etcd is the only persistent store in the control plane. All components are stateless except etcd. This is why etcd backups are so important — losing etcd without a backup means losing all cluster state.
Wrong. The controller manager has no database — it is stateless. It reads from and writes to the API server only.

3. After the scheduler assigns a Node to a Pod, what does the scheduler do next to start the container?

Wrong. The scheduler never contacts Nodes directly. It only talks to the API server.
Wrong. The scheduler knows nothing about the container runtime. The kubelet handles the container runtime interaction.
Correct! The scheduler only writes one thing: the spec.nodeName field on the Pod object, via the API server. That's it. The kubelet on that Node is watching the API server, notices the Pod is now assigned to it, and takes it from there.
Wrong. No component notifies another component directly. All communication goes through the API server. This is what makes the system loosely coupled and resilient.

4. Your readiness probe is failing on a Pod. The liveness probe is still passing. What happens?

Wrong. Container restarts are triggered by the liveness probe failing, not the readiness probe.
Correct! The readiness probe controls whether a Pod receives traffic from a Service. When it fails, the kubelet updates the Pod's Ready condition to False, the Endpoints controller removes the Pod IP from the Service's Endpoints, and kube-proxy updates the iptables rules. The container keeps running — it just gets no new requests until it becomes Ready again.
Wrong. Eviction is triggered by Node pressure (out of memory, disk) or Node failure — not probe failure.
Wrong. The HPA scales based on metrics (CPU, memory, custom) — not on probe status.
